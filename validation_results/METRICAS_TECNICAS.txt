================================================================================
DEFINIÇÃO FORMAL DA MATRIZ DE CONFUSÃO 2x2
================================================================================

Uma matriz de confusão 2x2 compara duas dimensões:

┌─────────────────────────────────┬──────────────────┬──────────────────┐
│                                 │ DETECTADO = SIM  │ DETECTADO = NÃO  │
├─────────────────────────────────┼──────────────────┼──────────────────┤
│ ESPERADO (Oráculo) = SIM        │ TP               │ FN               │
│                                 │ (Verdadeiro      │ (Falso Negativo) │
│                                 │  Positivo)       │                  │
├─────────────────────────────────┼──────────────────┼──────────────────┤
│ ESPERADO (Oráculo) = NÃO        │ FP               │ TN               │
│                                 │ (Falso Positivo) │ (Verdadeiro      │
│                                 │                  │  Negativo)       │
└─────────────────────────────────┴──────────────────┴──────────────────┘

Onde:
  - ESPERADO = Fragilidade definida no oráculo (ground truth)
  - DETECTADO = Fragilidade detectada pelo sistema
  - TP = Concordância correta (ambos detectam a fragilidade)
  - TN = Concordância correta (ambos não detectam a fragilidade)
  - FP = Discordância (sistema detecta, oráculo não prevê)
  - FN = Discordância (oráculo prevê, sistema não detecta)

================================================================================
FÓRMULAS DE CÁLCULO
================================================================================

1. ACURÁCIA (Accuracy)
   ────────────────────
   Definição: Proporção de predições corretas (TP + TN) sobre o total

   Fórmula:

       TP + TN
   ───────────────────────
   TP + FP + FN + TN

   Intervalo: [0, 100]%

   Interpretação:
   • 100%: Sistema perfeito
   • 90%:  Sistema muito bom
   • 75%:  Sistema aceitável
   • 50%:  Sistema não melhor que aleatório
   • 0%:   Sistema totalmente errado

   Limitação: Pode ser enganosa se há desbalanceamento entre positivos e negativos.
   Exemplo: Se 99% das fragilidades não existem, modelo que sempre diz "não"
            terá 99% de acurácia mas 0% de recall (péssimo em LGPD).

   Meta LGPD: ≥ 90%

────────────────────────────────────────────────────────────────────────────

2. PRECISÃO (Precision) - Also called "Positive Predictive Value"
   ──────────────────────────────────────────────────────────────
   Definição: De todas as detecções positivas do sistema, quantas estavam corretas?

   Fórmula:

          TP
   ──────────────
    TP + FP

   Intervalo: [0, 100]%

   Interpretação:
   • 100%: Todas as detecções foram corretas (0 falsos alarmes)
   • 90%:  9 em 10 detecções estão corretas
   • 80%:  8 em 10 detecções estão corretas
   • 50%:  Metade das detecções são falsas
   • 0%:   Todas as detecções são falsas

   Contexto LGPD:
   • Alta precisão: Sistema é confiável, poucos falsos alarmes
   • Baixa precisão: Sistema gera muitos "falsos positivos"
     Consequência: Rejeita pessoas conformes, prejudica experiência do usuário

   Quando usar: Quando você quer evitar falsos alarmes

   Problema com métrica sozinha: Um sistema que nunca detecta nada
   tem precisão indefinida (0/0). Por isso usamos F1-Score.

   Meta LGPD: ≥ 85%

────────────────────────────────────────────────────────────────────────────

3. RECALL (Sensitivity / True Positive Rate)
   ──────────────────────────────────────────
   Definição: De todas as fragilidades esperadas, quantas o sistema detectou?

   Fórmula:

          TP
   ──────────────
    TP + FN

   Intervalo: [0, 100]%

   Interpretação:
   • 100%: Sistema detectou todas as fragilidades esperadas (0 falsos negativos)
   • 90%:  Sistema detectou 9 de 10 fragilidades
   • 80%:  Sistema detectou 8 de 10 fragilidades
   • 50%:  Sistema detectou metade das fragilidades
   • 0%:   Sistema não detectou nenhuma fragilidade

   Contexto LGPD (CRÍTICO):
   • Alto recall: Você encontra todos os problemas de conformidade
   • Baixo recall: Você deixa passar fragilidades (MUITO RUIM em LGPD)
     Consequência: Conformidade falsa, risco de multas, perdas de dados

   Por quê é crítico em LGPD?
   Exemplo: Se seu sistema tem 80% de recall para violações de dados:
   • Em 100 casos reais, você detecta 80 e perdeu 20
   • Isso significa que 20% dos riscos passam desapercebidos
   • Uma violação perdida pode custar R$50M+ em multa LGPD

   Meta LGPD: ≥ 85% (Mais importante que Precisão)

────────────────────────────────────────────────────────────────────────────

4. F1-SCORE (Harmonic Mean of Precision and Recall)
   ──────────────────────────────────────────────────
   Definição: Balanço entre Precisão e Recall usando média harmônica

   Fórmula:

               2 × Precision × Recall
   F1-Score = ─────────────────────────
              Precision + Recall

   Ou simplificado:

                  2 × TP
   F1-Score = ──────────────────
              2×TP + FP + FN

   Intervalo: [0, 100]%

   Por que média harmônica? Porque:
   • Média aritmética de 100% e 0% = 50% (enganoso)
   • Média harmônica de 100% e 0% = 0% (correto: se uma é 0, resultado é 0)

   Interpretação:
   • 100%:   Perfeição (100% precision E 100% recall)
   • 90%:    Excelente balanço
   • 80%:    Bom balanço
   • 70%:    Aceitável mas não ideal
   • <70%:   Desbalanceamento grave entre P e R

   Quando usar: Quando você quer não sacrificar nenhuma das duas dimensões

   Contexto LGPD: Ideal porque:
   • Você não quer muitos falsos alarmes (precisão alta)
   • Você não quer perder fragilidades (recall alto)
   • F1-Score força o balanço

   Meta LGPD: ≥ 80% (Ideal: ≥ 85%)

────────────────────────────────────────────────────────────────────────────

5. ESPECIFICIDADE (Specificity / True Negative Rate)
   ───────────────────────────────────────────────────
   Definição: De todas as não-fragilidades, quantas foram corretamente rejeitadas?

   Fórmula:

          TN
   ──────────────
    TN + FP

   Intervalo: [0, 100]%

   Interpretação:
   • 100%: Sistema nunca detecta falso positivo
   • 90%:  Sistema rejeita 90% dos casos sem fragilidade corretamente
   • 80%:  Sistema rejeita 80% dos casos sem fragilidade corretamente
   • 50%:  Metade das rejeições são erradas
   • 0%:   Sistema sempre detecta falso positivo

   Relação com Precisão:
   • Especificidade foca em TN (casos negativos)
   • Precisão foca em TP (caso positivos detectados)
   • Ambas usam FP mas em contextos diferentes

   Contexto LGPD:
   • Alta especificidade: Poucos falsos positivos, sistema confiável
   • Baixa especificidade: Muitos falsos alarmes, afasta usuários

   Meta LGPD: ≥ 75% (Menos crítico que Recall, mas importante)

================================================================================
COMPARAÇÃO DE MÉTRICAS
================================================================================

Métrica       Foca em    Quando Usar               Crítica em LGPD?
────────────────────────────────────────────────────────────────────
Acurácia      Tudo       Visão geral               Sim - filtro inicial
Precisão      FP/TP      Reduzir falsos alarmes   Sim - confiabilidade
Recall        FN/TP      Encontrar tudo           ✓✓ CRÍTICA - risco
Especificidade TN/FP    Rejeitar corretamente    Não - secundária
F1-Score      P + R      Balanço                  Sim - recomendado

Hierarquia de Importância em LGPD:
  1. RECALL (Crítica) - Não pode deixar passar riscos
  2. F1-Score (Crítica) - Balanço entre risco e confiabilidade
  3. Precisão (Alta) - Evita desconfiança dos usuários
  4. Acurácia (Média) - Validação geral
  5. Especificidade (Baixa) - Nice-to-have

================================================================================
EXEMPLOS NUMÉRICOS DETALHADOS
================================================================================

EXEMPLO 1: Bom Desempenho
───────────────────────
TP = 90, FP = 10, FN = 10, TN = 90, Total = 200

Acurácia:     (90+90)/200 = 90%      ✓
Precisão:     90/(90+10) = 90%       ✓
Recall:       90/(90+10) = 90%       ✓
F1-Score:     2×(90×90)/(90+90) = 90% ✓
Especificidade: 90/(90+10) = 90%     ✓

Diagnosis: EXCELENTE - Sistema está perfeitamente balanceado

───────────────────────

EXEMPLO 2: Alta Precisão, Baixo Recall (Conservador)
─────────────────────────
TP = 45, FP = 5, FN = 55, TN = 95, Total = 200

Acurácia:     (45+95)/200 = 70%      ✗ Baixa
Precisão:     45/(45+5) = 90%        ✓ Alta
Recall:       45/(45+55) = 45%       ✗ MUITO BAIXA (CRÍTICO)
F1-Score:     2×(90×45)/(90+45) = 60% ✗ Ruim
Especificidade: 95/(95+5) = 95%      ✓ Excelente

Diagnosis: PÉSSIMO para LGPD
Problema: Sistema é muito conservador, deixa passar 55 fragilidades
Consequência: Conformidade falsa, risco de violações não detectadas
Ação: REJEITAR este sistema - recall baixo demais

───────────────────────

EXEMPLO 3: Baixa Precisão, Alto Recall (Agressivo)
──────────────────────────
TP = 95, FP = 55, FN = 5, TN = 45, Total = 200

Acurácia:     (95+45)/200 = 70%      ✗ Baixa
Precisão:     95/(95+55) = 63.3%     ✗ Baixa (muitos FP)
Recall:       95/(95+5) = 95%        ✓ Excelente
F1-Score:     2×(63.3×95)/(63.3+95) = 76.3% ⚠ Borderline
Especificidade: 45/(45+55) = 45%     ✗ Baixa

Diagnosis: ACEITÁVEL mas com ressalvas
Benefício: Quase todas as fragilidades são detectadas (95%)
Custo: Muitos falsos alarmes (55 detecções falsas)
Tradeoff: Preferível em LGPD (melhor deixar extras do que perder reais)
Ação: ACEITAR se F1-Score ≥ 80%, caso contrário refinarm

───────────────────────

EXEMPLO 4: Balanceado Ruim (Resultado Decepcionante)
─────────────────────────
TP = 60, FP = 30, FN = 40, TN = 70, Total = 200

Acurácia:     (60+70)/200 = 65%      ✗ Abaixo
Precisão:     60/(60+30) = 66.7%     ✗ Abaixo
Recall:       60/(60+40) = 60%       ✗ ABAIXO (crítico)
F1-Score:     2×(66.7×60)/(66.7+60) = 63% ✗ Ruim
Especificidade: 70/(70+30) = 70%     ✗ Abaixo

Diagnosis: FALHA CRÍTICA - Nenhuma métrica aceita
Problema: Sistema não funciona bem em nenhuma dimensão
Ação: REJEITAR completamente, fazer redesign completo

================================================================================
MATRIZ DE DECISÃO DE ACEITAÇÃO
================================================================================

Recall < 80%
├─→ ✗ REJEITAR IMEDIATAMENTE
│   (Perdendo muitas fragilidades)
│
└─ Recall ≥ 80%
   │
   ├─ Precisão < 60%
   │  ├─ F1-Score < 70%
   │  │  └─→ ⚠ REVISAR (muitos FP, mas recall ok)
   │  │
   │  └─ F1-Score ≥ 70%
   │     └─→ ✓ ACEITAR COM CAUTELA
   │        (Trade-off FP/recall aceitável)
   │
   └─ Precisão ≥ 80%
      ├─ F1-Score < 80%
      │  └─→ ⚠ REVISAR (bom mas não ideal)
      │
      └─ F1-Score ≥ 80%
         ├─ Acurácia ≥ 85%
         │  └─→ ✓✓ ACEITAR (Pronto para Produção)
         │
         └─ Acurácia < 85%
            └─→ ⚠ REFINAR (Muito próximo, testar mais)

CENÁRIOS REAIS DE APLICAÇÃO
================================================================================

CENÁRIO 1: Validação de Persona P01
───────────────────────────────────
Teste: Persona responde o questionário 10 vezes (simulações independentes)
Resultado agregado:

Detecções esperadas (oráculo): F1, F3, F9
Detecções do sistema (média das 10 rodadas):
  - F1 detectado: 9/10 vezes ✓
  - F3 detectado: 8/10 vezes ⚠
  - F9 detectado: 7/10 vezes ⚠
  - Falsos positivos: 2/10 vezes
  - Falsos negativos: 3/10 vezes

Cálculo de TP, FP, FN, TN:
  Total de fragilidades testadas = 10 simulações × 3 fragilidades = 30 esperadas
  TP = 9+8+7 = 24 (detectou 24 das 30)
  FP = 2 (2 falsos alarmes)
  FN = 3+2+3 = 8 (não detectou 8 das 30) - obs: soma de não-detectados
  ...detalhamento mais complexo

Obs: Na prática, cada simulação gera múltiplos valores (perguntas × personas)
e a agregação é mais sofisticada. Este é um exemplo simplificado.

================================================================================
IMPLEMENTAÇÃO NO COMPONENTE
================================================================================

O componente ConfusionMatrix.tsx calcula automaticamente:

```typescript
const metrics = useMemo(() => {
  const { tp, fp, fn, tn } = data;
  const total = tp + fp + fn + tn;

  // Evita divisão por zero
  const accuracy = total > 0 ? ((tp + tn) / total * 100) : 0;
  const precision = (tp + fp) > 0 ? (tp / (tp + fp) * 100) : 0;
  const recall = (tp + fn) > 0 ? (tp / (tp + fn) * 100) : 0;

  // F1-Score com tratamento de casos limítrofes
  const f1Score = (precision + recall) > 0
    ? (2 * (precision * recall) / (precision + recall))
    : 0;

  const specificity = (tn + fp) > 0 ? (tn / (tn + fp) * 100) : 0;

  return {
    accuracy: Math.round(accuracy * 10) / 10,      // 1 casa decimal
    precision: Math.round(precision * 10) / 10,
    recall: Math.round(recall * 10) / 10,
    f1Score: Math.round(f1Score * 10) / 10,
    specificity: Math.round(specificity * 10) / 10,
  };
}, [data]);
```

Nota: Arredondamento para 1 casa decimal para legibilidade.
