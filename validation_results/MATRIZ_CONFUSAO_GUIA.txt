================================================================================
ESTRUCTURA DE DADOS
================================================================================

ConfusionMatrixData {
  truePositives: number;      // TP: Esperado=SIM, Detectado=SIM ✓ Correto
  falsePositives: number;     // FP: Esperado=NÃO, Detectado=SIM ✗ Falso Alarme
  falseNegatives: number;     // FN: Esperado=SIM, Detectado=NÃO ✗ Perdido
  trueNegatives: number;      // TN: Esperado=NÃO, Detectado=NÃO ✓ Correto
}

================================================================================
MÉTRICAS CALCULADAS
================================================================================

1. ACURÁCIA (Accuracy)
   Fórmula: (TP + TN) / (TP + FP + FN + TN) * 100
   O quê: Proporção total de previsões corretas (positivas e negativas)
   Quando usar: Visão geral de desempenho quando positivos e negativos são balanceados
   Meta LGPD: ≥ 90%

2. PRECISÃO (Precision)
   Fórmula: TP / (TP + FP) * 100
   O quê: De todas as detecções feitas, quantas estavam corretas
   Quando usar: Você quer evitar falsos alarmes (FP)
   Importância LGPD: Alta - evita acusar conforme quando não é
   Meta: ≥ 85%

3. RECALL / SENSIBILIDADE (Recall)
   Fórmula: TP / (TP + FN) * 100
   O quê: De todas as fragilidades esperadas, quantas foram detectadas
   Quando usar: Você quer encontrar todos os problemas (evitar FN)
   Importância LGPD: Crítica - não perder fragilidades reais
   Meta: ≥ 85%

4. F1-SCORE
   Fórmula: 2 × (Precision × Recall) / (Precision + Recall)
   O quº: Média harmônica entre Precisão e Recall
   Quando usar: Balanceamento entre evitar FP e FN
   Importância LGPD: Crítica - balanço entre segurança e confiabilidade
   Meta: ≥ 80%

5. ESPECIFICIDADE (Specificity)
   Fórmula: TN / (TN + FP) * 100
   O quê: Capacidade de identificar corretamente casos negativos (sem fragilidade)
   Quando usar: Você quer evitar falsos alarmes
   Importância LGPD: Média - reduz negativas falsas

================================================================================
ESTRUTURA DAS PERSONAS
================================================================================

Cada persona tem:

1. ARQUIVO MARKDOWN (.md)
   Localização: personas/P<XX>_<Nome>.md
   Conteúdo:
   - Identidade: setor, cargo, nível hierárquico
   - Contexto: rotina diária, responsabilidades
   - Fragilidades: comportamentos problemáticos

2. ARQUIVO ORÁCULO (YAML)
   Localização: oraculos/P<XX>_<Nome>_oraculo.yml
   Conteúdo:
   - persona_id: Identificador único
   - nivel_risco_esperado: alto/médio/baixo
   - fragilidade_central: Descrição principal
   - vetor_de_fragilidade: Intensidade 0-3 para cada eixo (F1-F8)
   - categorias_lgpd_esperadas: Fragilidades específicas esperadas
   - comportamentos_observáveis: Como a persona deveria se comportar
   - nao_deve_ser_detectado: Falsos positivos a evitar
   - criterio_de_sucesso: Definição de sucesso da validação

================================================================================
EIXOS DE FRAGILIDADE (F1-F8)
================================================================================

F1: Compartilhamento Informal
    Descrição: Dados compartilhados por canais não autorizados (WhatsApp, email pessoal)
    Exemplo: Enviar currículo por WhatsApp

F2: Armazenamento Pessoal / Indevido
    Descrição: Dados mantidos em locais não autorizados
    Exemplo: Manter planilha própria com dados de clientes

F3: Retenção Excessiva
    Descrição: Dados mantidos além do necessário
    Exemplo: Guardar currículos de candidatos rejeitados indefinidamente

F4: Coleta Excessiva
    Descrição: Coleta de mais dados do que necessário
    Exemplo: Pedir informações de saúde não relacionadas ao cargo

F5: Acesso Excessivo
    Descrição: Acesso a dados além do necessário para o cargo
    Exemplo: Vendedor tendo acesso a todos os clientes, não apenas seus

F6: Falta de Transparência
    Descrição: Usuários não sabem como seus dados são usados
    Exemplo: Não informar sobre análise de padrões de compra

F7: Uso Secundário
    Descrição: Dados usados para fins diferentes do original
    Exemplo: Usar dados de contato para marketing não consentido

F8: Terceiros Sem Controle
    Descrição: Compartilhamento com terceiros sem devida supervisão
    Exemplo: Enviar dados para consultores externos sem contrato

F9: Dados Sensíveis sem Salvaguarda
    Descrição: Dados sensíveis (saúde, raça, etc.) sem proteção extra
    Exemplo: Armazenar informação de deficiência em pasta comum

F10: Incidente Mal Tratado
     Descrição: Vazamento ou perda não reportado corretamente
     Exemplo: Descobrir vazamento mas não comunicar à liderança

EXEMPLOS DE VALIDAÇÃO
================================================================================

EXEMPLO 1: P01 - RH Recrutamento
───────────────────────────────
Persona: Analista de Recrutamento que compartilha currículos por WhatsApp

Fragilidades Esperadas (Oráculo):
  - F1: Compartilhamento informal (Severidade: Alta)
  - F3: Retenção excessiva (Severidade: Moderada)
  - F9: Dados sensíveis sem salvaguarda (Severidade: Alta)

Matriz de Confusão (Exemplo):
  VP: 8  (Detectou F1, F3, F9 corretamente)
  FP: 2  (Detectou incorretamente: Cobrança, Birô de crédito)
  FN: 1  (Perdeu detectar: Um detalhe de F3)
  VN: 14 (Corretamente não detectou: Compartilhamento com processadores, etc)

Cálculos:
  Acurácia: (8 + 14) / 25 = 88%        ⚠ Abaixo da meta 90%
  Precisão: 8 / 10 = 80%                ⚠ Abaixo da meta 85%
  Recall: 8 / 9 = 88.9%                 ✓ Acima da meta
  F1-Score: 2×(80×88.9)/(80+88.9) = 84.2% ⚠ Abaixo da meta 85%

Diagnosis:
  - Sistema detectou fragilidades corretas (bom recall)
  - Mas gerou falsos alarmes (FP baixa)
  - Falta refinamento para reduzir FP

───────────────────────────────

EXEMPLO 2: A01 - Consultor RH Malicioso
───────────────────────────────
Persona: Consultor que viola LGPD deliberadamente

Fragilidades Esperadas (Oráculo):
  - Múltiplas violações intencionais

Matriz de Confusão (Exemplo):
  VP: 12 (Detectou corretamente todas as violações)
  FP: 0  (Sem falsos alarmes)
  FN: 0  (Não perdeu nada)
  VN: 13 (Corretamente rejeitou não-violações)

Cálculos:
  Acurácia: (12 + 13) / 25 = 100%       ✓ Perfeito
  Precisão: 12 / 12 = 100%              ✓ Perfeito
  Recall: 12 / 12 = 100%                ✓ Perfeito
  F1-Score: 100%                        ✓ Perfeito

Diagnosis:
  - Sistema funcionando perfeitamente para casos de violação óbvia
  - Pronto para produção

================================================================================
CRITÉRIOS DE ACEITAÇÃO PARA PRODUÇÃO
================================================================================

Métrica                Mínimo      Ideal       Crítica?
────────────────────────────────────────────────────────
Acurácia               85%         ≥ 90%       Sim
Precisão               80%         ≥ 85%       Sim
Recall                 85%         ≥ 90%       Sim (NÃO perder fragilidades)
F1-Score               80%         ≥ 85%       Sim
Especificidade         75%         ≥ 85%       Não (menos crítico)

Obs: O Recall é crítico em LGPD porque perder uma fragilidade real é pior
que ter um falso alarme.

================================================================================
COMO USAR O COMPONENTE
================================================================================

1. IMPORTAR O COMPONENTE

   import { ConfusionMatrix } from "@/components/ConfusionMatrix";
   import type { ConfusionMatrixData } from "@/components/ConfusionMatrix";

2. PREPARAR OS DADOS

   const confusionData: ConfusionMatrixData = {
     truePositives: 8,
     falsePositives: 2,
     falseNegatives: 1,
     trueNegatives: 14,
   };

3. RENDERIZAR

   <ConfusionMatrix
     data={confusionData}
     title="Validação Persona P01"
     description="Comparação entre Oráculo (Esperado) e Sistema (Detectado)"
   />

4. VISUALIZAR MÉTRICAS AUTOMÁTICAS

   O componente calcula automaticamente:
   - Acurácia
   - Precisão
   - Recall
   - F1-Score
   - Especificidade
